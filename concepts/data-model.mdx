This page describes the shape of the core objects you'll encounter when working with the Olira API. Every response from Olira is JSON, and each object type has a consistent set of fields you can rely on across all endpoints. Familiarising yourself with these structures will help you write cleaner parsing and validation logic in your application.

## Resource object

A Resource is the primary entity in Olira. You create and manage resources through the API, and all other objects — events, webhooks — are anchored to them.

```json
{
  "id": "res_01HXYZ",
  "name": "My Resource",
  "status": "active",
  "metadata": {},
  "created_at": "2024-01-15T10:30:00Z",
  "updated_at": "2024-01-15T10:30:00Z"
}
```

<ResponseField name="id" type="string" required>
  The unique identifier for the resource. All IDs in Olira are UUIDs and are assigned by the server at creation time — you cannot supply your own.
</ResponseField>

<ResponseField name="name" type="string" required>
  A human-readable display name for the resource. Must be non-empty. You can update this field at any time with a `PATCH` request.
</ResponseField>

<ResponseField name="status" type="string" required>
  The current lifecycle state of the resource. Possible values are `active` and `inactive`. Newly created resources default to `active`.
</ResponseField>

<ResponseField name="metadata" type="object">
  A free-form object for storing custom key-value pairs alongside the resource. Keys must be strings; values may be strings, numbers, or booleans. Olira does not interpret or index metadata fields by default.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  The timestamp at which the resource was created, in ISO 8601 UTC format (e.g. `2024-01-15T10:30:00Z`). Set by the server and never changes.
</ResponseField>

<ResponseField name="updated_at" type="string" required>
  The timestamp at which the resource was last modified, in ISO 8601 UTC format. Updated automatically by the server whenever a write operation succeeds.
</ResponseField>

## Event object

Events are immutable records generated by Olira whenever a resource changes state. You can query events directly or receive them in real time through webhooks. Because events are never modified or deleted, they provide a reliable audit log.

```json
{
  "id": "evt_01HABC",
  "type": "resource.created",
  "resource_id": "res_01HXYZ",
  "data": {},
  "created_at": "2024-01-15T10:30:00Z"
}
```

<ResponseField name="id" type="string" required>
  The unique identifier for the event. Event IDs are UUIDs prefixed with `evt_`.
</ResponseField>

<ResponseField name="type" type="string" required>
  A dot-separated string describing what occurred, following the `resource.action` convention. Common values include `resource.created`, `resource.updated`, and `resource.deleted`. You can use this field to filter events when querying the log or configuring webhook subscriptions.
</ResponseField>

<ResponseField name="resource_id" type="string" required>
  The `id` of the Resource this event is associated with. Use this field to correlate events back to their source objects.
</ResponseField>

<ResponseField name="data" type="object">
  A snapshot of the relevant resource data at the time the event was generated. The exact shape of this object depends on the event `type`. For update events, `data` may include both `previous` and `current` sub-objects so you can diff the change.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  The timestamp at which the event was recorded, in ISO 8601 UTC format. Because events are immutable, this value never changes after creation.
</ResponseField>

<Note>
  The `created_at` on an event reflects when Olira recorded the state change, which may differ slightly from when you receive the event via webhook due to delivery latency.
</Note>

## Webhook object

A Webhook defines a subscription that tells Olira where to deliver event payloads and which event types to include. You can create and manage webhooks through the API, and Olira will `POST` matching event payloads to the configured `url` in real time.

```json
{
  "id": "wh_01HDEF",
  "url": "https://your-server.com/webhooks/olira",
  "events": ["resource.created", "resource.updated"],
  "active": true,
  "created_at": "2024-01-15T10:30:00Z"
}
```

<ResponseField name="id" type="string" required>
  The unique identifier for the webhook. Webhook IDs are UUIDs prefixed with `wh_`.
</ResponseField>

<ResponseField name="url" type="string" required>
  The fully qualified HTTPS URL to which Olira will deliver event payloads. Must be publicly reachable and use HTTPS. You can update this field at any time.
</ResponseField>

<ResponseField name="events" type="string[]" required>
  An array of event type strings this webhook is subscribed to (e.g. `["resource.created", "resource.updated"]`). Olira only delivers events whose `type` appears in this list. Pass `["*"]` to subscribe to all event types.
</ResponseField>

<ResponseField name="active" type="boolean" required>
  Whether this webhook is currently enabled. When `false`, Olira will not deliver events to the endpoint. You can toggle this field to pause and resume delivery without deleting the webhook.
</ResponseField>

<ResponseField name="created_at" type="string" required>
  The timestamp at which the webhook was registered, in ISO 8601 UTC format.
</ResponseField>

<Tip>
  If your endpoint returns a non-2xx status code, Olira will retry delivery with exponential backoff. Set `active` to `false` if you need to take your endpoint offline for maintenance so the retry queue doesn't fill up.
</Tip>
